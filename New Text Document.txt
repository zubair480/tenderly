You are building the frontend for "Tenderly", a hackathon project due TODAY at 5pm PDT. Tenderly is an AI volunteer-matching platform for San Francisco: a volunteer uploads a resume, AI builds their profile, and the app proactively recommends the 3 nonprofit opportunities where they'd make the biggest difference, each with a personal AI-written explanation. A live "community needs" layer from SF open data re-ranks recommendations when conditions change.

We are explicitly targeting the "Best UI/UX" prize. Judging criteria: intuitive interaction, accessibility, readability, best practices. The design must feel warm and human (this is social good, not a SaaS dashboard) while staying crisp.

Build a React + Vite + TypeScript single-page app with Tailwind. No auth. The backend is being built in parallel; code against the contract below and use a mock layer first.

## API contract (do NOT change shapes; backend implements these exactly)
Base URL from env VITE_API_URL, default http://localhost:8080

POST /api/profile (multipart form: file, interests, availability)
-> { "profile_id", "name", "skills": [], "experience_summary", "causes": [], "availability" }

GET /api/matches/{profile_id}?scenario=normal|surge
-> { "matches": [ { "opportunity_id", "org_name", "title", "category", "neighborhood", "lat", "lng", "commitment", "score" (0-1 float), "urgency" ("low"|"medium"|"high"), "why_you" (string or null) } ], "scenario", "needs_summary" }

GET /api/needs
-> { "updated_at", "neighborhoods": [ { "name", "case_count", "top_categories": [] } ] }

## Build src/api/mock.ts first with realistic fake responses (including a 1.5s artificial delay) and a VITE_USE_MOCK flag, so the full UI is demoable before backend integration. Then src/api/client.ts for the real calls.

## Screens / flow (one continuous scroll-or-step flow, no router needed)

1. Landing / onboarding
- Hero: one strong sentence about the problem ("Millions want to help. Tenderly finds where YOU matter most.") and a single CTA.
- Onboarding card: drag-and-drop resume upload (PDF/TXT), interest chips (multi-select from ~10 causes), availability selector (segmented control). One screen, no wizard fatigue.

2. Profile reveal
- After upload, show an engaging loading state (rotating messages like "Reading your experience...", "Scanning what SF needs right now...") because the AI call takes a few seconds.
- Then an animated reveal of their AI profile: name, skill chips, one-line experience summary, causes. Feels like a moment, not a form response.

3. Matches (the core screen)
- Top 3 as rich cards: org, title, neighborhood, commitment, urgency badge (color + label, never color alone), a match strength ring/bar showing score as a percentage, and the "why_you" text featured prominently in a quote-style block. This explanation is our differentiator; give it visual weight.
- Ranks 4+ as a compact list below ("Also good fits").
- A "Community pulse" sidebar/section fed by /api/needs: which neighborhoods have the most activity, updated_at timestamp, and the needs_summary sentence.
- THE DEMO MOMENT: a prominent toggle or button "Simulate: cold snap hits SF" that refetches with scenario=surge. Cards must animate their re-ordering (framer-motion layout animations) so judges SEE recommendations adapt live. Include a small banner explaining what changed.

4. Simple map (nice-to-have, only if time remains): pins for the top matches using react-leaflet with OpenStreetMap tiles. Skip if it threatens the deadline.

## Design system
- Warm palette: cream/off-white background, deep green or warm terracotta primary, high contrast text (WCAG AA minimum, check contrast ratios).
- One display font + one body font from Google Fonts (e.g. Fraunces + Inter). Generous whitespace, max content width ~1100px, mobile-responsive.
- Micro-interactions: hover states, staggered card entrance, the re-rank animation. Subtle, 200-300ms, no gimmicks.
- Accessibility: semantic HTML, labels on all inputs, focus-visible rings, keyboard operable upload (also a click-to-browse fallback), aria-live region announcing when matches update, prefers-reduced-motion respected.
- Empty/error states: friendly message + retry button for any failed fetch. The app must never show a blank screen.
